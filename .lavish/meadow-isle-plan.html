Meadow Isle — Development Plan

FINAL · 2026-07-17 · reviewed with Buddy design authority: docs/islands/meadow-isle.md

Turn the first creature island from a design doc into a playable, curriculum-true Standard-1 experience: walkable island, catchable creatures, and a question engine that asks like a real SJKC worksheet — counting sheep, reading clocks, counting coins.

Design authority: docs/islands/meadow-isle.md · Curriculum: docs/curriculum/standard-1-sjkc-math.md + standard-1-question-style.md · World: docs/world-map-design.md

Where we stand (verified against the repo, 2026-07-17)

✅ World rendering

game/assets/src/world/ — tile-map format (MAP: string[], 48px tiles, collision, dock tiles) proven by Harbor Town. Meadow Isle reuses the same format.

✅ Battle + shop loops

BattleScreen.ts, QuestionView.ts, shared/battle-rules.ts, shared/shop-rules.ts — turn-based battle and a shop exist.

✅ Art pipeline

remote-art.ts already serves the licensed Pocket Creature Tamer pack from private R2 (/art/v1/…). The 12-family Meadow roster can render as real sprites — no new infra.

⚠️ Question engine is Year-4 shaped

shared/question-engine.ts = numeric answer + makeChoices() near-misses. Std 1 needs visual formats, non-numeric answers, TP levels, profile gating. This is the long pole.

⚠️ Stats live on the creature

shared/creature.ts stores HP/attack/level/XP per creature. World-map direction says the player owns stats. A migration is owed; timing is Decision D2 below.

✅ Curriculum ground truth

Two research docs pin scope + formats with real-paper evidence and a validated schema (§E). The generator prompt is mostly written already.

What we're building

Each landmark anchors one Std-1 topic; counting & + − appear everywhere. Cognitive demand (TP level) rises with distance from the dock — formats stay free to roam. Shape: a big explorable ring road built as a region graph — named-gateway chunks, never resized, with expansion-pocket gateways reserved for future rest stops & mini-games.

Milestones — each ends playable

  • M1 — Sail there & walk it
    Harbor-master NPC at the existing dock (D tiles) sails you out. Meadow Isle is a region graph, not one big rectangle: each area is its own MAP: string[] chunk with stable local coordinates, connected by named gateways in a ring — never resized after shipping. An expansion pocket is a gateway pointing at a not-yet-existing region; future rest stops / mini-games attach as new chunks on the inner or outer ring without moving any existing geography, NPC, or save coordinate. Regions are still bigger than one screen; camera follows the player, clamped to the current region; region banners on crossing ("Woolly Meadows 羊毛草原"). Location persistence (regionId + local x/y, with a load-time walkable check and fallback to the region's safe gateway) lands with the save-v2 PR before M2 — M1 stays session-only like today, so the island is never blocked on the migration. Placeholder creature art, signpost shortcuts to the dock, exit ferry home.
    Touches: new world/regions/ registry (one stable file per region + named gateway definitions), GameApp.ts routing, state.ts
    Playable: walk the whole island
  • M2 — Real creatures, real encounters
    12-family roster into species data keyed by pack ID; slice the 144×48 strips into 48×48 stage frames via the existing R2 loader; per-area encounter tables + rarity; _alt as special-colour rares.
    Two gates before encounters may ship
    • Curriculum-compliant placeholder bank. Today's SAMPLE_BANK is Year-4 content (40 × 320, RM55,000) — it must not be heard on Meadow Isle even as a placeholder. M2 includes a hand-authored ~20-item legacy-numeric bank that obeys every island law (≤100, + − only, ≤RM10, single-step), verified by the same mechanical checks M4 will automate.
    • Full-team capture rule. The team (max 6) is the whole collection for now — so when it's full, capture is blocked with a gentle message ("Your team is full! Make a friend at home first.") rather than pushing a 7th creature into an invalid save. Acceptance test: a capture attempt at 6/6 leaves the save valid and unchanged. Box storage / releasing is a later-island feature.
    • Creature identity in save v2. Real sprites/evolution/variants need speciesId (pack ID), stage, and variant on each creature. Lands with the D2 migration PR (below), which maps legacy creatures to legacy/<name> species, stage 1, normal variant.
    Touches: shared/creature.ts (SPECIES), remote-art.ts (frame slicing), encounter config per area, placeholder bank
    Playable: wild battles with real sprites across the island
  • M3 — Std-1 question schema long pole
    Extend question-engine.ts to the §E schema: format_type, tp_level, item_format, presentation, answer_form (numeral · circle · ordering · match · chinese-word …), distractor_strategy, bilingual {numeral, zh_word}. Numeric answers become one answer form, not the only one. Domain tests first.
    Compatibility strategy (keeps battle & shop playable through M2–M5)
    • Versioned schema, no in-place rewrite. Legacy Question (numeric, snake_case bank format) stays byte-for-byte untouched: absent schema (or schema: 1) means legacy; only new items carry the schema: 2 discriminator. Banks may mix both, and the existing SAMPLE_BANK needs zero edits.
    • Adapters in shared/. asLegacyTurn(v2) degrades any V2 item whose answer_form is numeral to the existing QuestionTurn/makeChoices() path; toV2(q) upgrades authored v1 items. QuestionRound becomes a union wrapper — legacy rounds keep choices: number[] + judge(number); new forms carry their own graders.
    • QuestionView switches on answer_form, one sub-renderer per form, defaulting to today's numeric panel. Battle/shop code compiles and plays unchanged against v1 the whole time.
    • Rollback switch. The V2 path is gated by the save's profile + bank schema field — a bad V2 batch is disabled server-side, v1 never redeploys.
    Touches: shared/question-engine.ts, shared/question-bank.ts, shared/tests/, save-types.ts (profile flag)
    Playable: engine unit-tests every §A format and the legacy suite passes unmodified
  • M4 — Generator pipeline
    LLM generation constrained by the two curriculum docs (hard constraints as system rules; per-topic format weights; distractor strategies). Validation gate: schema check + constraint rejection (≤100, +− only, RM10, single-step) + adversarial verify pass, mirroring how the research corpus itself was verified. Approved items → D1. See Decision D1 (batch vs serve-time).
    Touches: tools/ (generator + validator), worker/ (bank API), D1 schema
    Playable: island encounters serve validated Std-1 items
  • M5 — Landmarks & figures
    The figure renderers + interactions that make questions visible: ten-frame sheep pens, settable clock-tower face, fruit stand (reuse shop UI, ≤RM10 catalog), festival pictograph board, marble balance, shape flowerbeds. Ship per presentation type, most-referenced first (ten-frame, clock, coins, objects → then balance, pictograph, abacus, calendar).
    Touches: QuestionView.ts figure components, landmark NPC scripts, shop-rules.ts catalog
    Playable: questions render as world objects, not prose
  • M6 — Guardian, badge, evolution
    Cloud-Maned Horse (Uniques/03) at the Hundred Stones: a multi-topic slate (count, +− sentence, clock read, money change, pictograph read), TP3–4 with one TP5 twist — fixed authored slate (D4). Victory → Meadow Badge in save data; the badge evolves the player's starter, located by starterCreatureId — no FOMO rule. (No box storage on Meadow Isle yet, so in practice: the team; the rule extends automatically when storage lands.) Gentle defeat back to the dock, nothing lost.
    Touches: boss config in battle, save-types.ts (badges), evolution table in creature.ts
    Playable: Meadow Isle is completable end-to-end
  • M7 — Progression tuning
    Level-relative XP falloff (island below your level → 0–1 XP), island stat curve so Tallgrass reads "lethal" at level 1 and Meadow reads "breeze" at level 10, catch rates, alt-colour rates. Numbers validated by playtesting with the target audience (two kids, ages 7 & 10, conveniently on file).
    Touches: battle-rules.ts, encounter/XP config; depends on Decision D2
    Playable: the stat curve is the gate

M1→M2→M3 is the critical path. M4 depends on M3's schema. M5 can overlap M4. M6 needs M4+M5. M7 is last because tuning needs everything real.

Decisions needed from you

D1 · Question generation timing

A · Offline batch → validate → D1 recommended

Generate in bulk via tools/, gate hard, ship approved items. Zero latency, cost-controlled, kid-safe by construction. Matches ROADMAP's "validation gate" instinct.

B · Generate at serve time

Infinite variety, but latency in a kid's battle, per-request cost, and validation must run inline on theWorker hot path.

C · Hybrid

Batch seed + background top-up when a topic×TP cell runs low. Best long-term; more moving parts now.

D2 · Stats migration timing

A · Migrate to player-owned stats before M2 recommended

World-map direction already chose this; doing it now means encounters, catching, and M7 tuning are built once on the final model. Small change today: creature.ts + battle-rules.ts + save migration (plan below).

Save migration plan (v1 → v2) — prerequisite for option A

Honest starting point: validateSaveState today rejects anything but version === 1, and SaveState has no player stats, badges, or curriculum profile. "Versioning exists" means a version field, not a migrator.

  • Upgrader, not rewriter. New migrateSave(raw) in shared/: accepts only well-formed v1 (version: 1 required — absent stays invalid, exactly as today), then validates-as-v1 → migrates → validates-as-v2. One-way, tested per version pair.
  • Strict decode, no pass-through. validateSaveState is a trust boundary: decode only known v1 fields and construct a fresh v2 object from them — arbitrary unknown fields are dropped at the boundary, never carried into the new schema. Every known v1 field has a defined home: team (names, colors, boss flags), money, bag, savedAt carry over verbatim; per-creature stat fields become derived-from-player going forward (world-map model) and are retained read-only for one version, then removed in v3.
  • Creature identity added in v2. Each creature gains speciesId, stage, variant, plus a stable per-instance creatureId; the save gains starterCreatureId (set to the v1 team's first creature — the professor's starter). Legacy creatures map to reserved legacy species (legacy/addlepuff, legacy/subtractopus, …) at stage 1, variant normal — they stay playable forever but never appear in island encounter tables. Box storage is out of scope for Meadow Isle: the team (max 6) is the whole collection until a later milestone adds storage; creatureId makes that migration safe when it lands.
  • Location added in v2. location: { regionId, x, y } — v1 saves migrate to the Harbor Town spawn. On every load the saved tile is validated walkable; if a layout edit made it solid, spawn falls back to that region's named safe gateway.
  • Concrete XP conversion. New player XP is derived so no child ever loses ground: playerXp := max over team of ((level − 1) × XP_PER_LEVEL + xp) (matching creature.ts's current per-creature accumulation; XP_PER_LEVEL = 20).
  • Rollout order. Worker deploys the accept-and-migrate reader first; the game client ships v2 writes only after. D1 stores saves as JSON payloads keyed per account, so migration happens at the read/write boundary — no DB schema change.
  • Acceptance tests. Fixture v1 saves (fresh createNewGame() + a mid-game example) must migrate, re-validate, round-trip, and boot into a playable battle. A corrupted or future-version save still fails cleanly.
B · Ship Meadow on creature-owned stats, migrate later

Faster M1–M2, but every tuning number in M7 gets computed twice, and catching "instantly usable pets" can't work as designed until the migration lands.

D3 · Answer-form scope for first playable

A · numeral + circle (objective) + ordering first recommended

Covers most of §A's high-frequency formats (count-write, compare, order-sequence, fill-blank, true-false) with UI we already half-have. match/color/tick follow in M5; drawing/trace-write are cut from the island entirely (see Risks).

B · All 10 answer forms at once

Tracing/drawing and colouring need bespoke touch UI; bundling them delays everything behind the hardest 20%.

D4 · Guardian question slate

A · Fixed authored slate v1 recommended

The guardian is a set piece — one hand-checked question per topic (count, +−, clock, money, pictograph). Deterministic, reviewable, no generator dependency for the island's one critical battle.

B · Mastery-weighted sampling

Targets each child's weakest topics — great pedagogy, but needs per-topic mastery tracking that doesn't exist yet. A later-island upgrade.

Risks

HighGenerator leaks out-of-level content. The research docs' own verify pass dropped real exam items for being multi-step — prompt wording alone won't hold the line.
Recommended Three-layer gate in M4, all in tools/validate-question.ts: (1) mechanical — parse the expression, hard-reject × ÷, any operand/result > 100, multi-step chains; money parser rejects > RM10 and mixed coin+note exchange; validator independently re-derives the stated answer and rejects mismatches; (2) adversarial — a second LLM call runs the corpus' own checklist (authentic / on-level / correctly-answered) and rejects, never edits; (3) human sample — you review the first 200 items 100%, then a 5% random sample per batch.
MediumFigure-render scope creep. 13 presentation types are each bespoke UI.
Recommended One FigureView kit, config not custom UI. Every figure is a small declarative spec (rows of icons, labels, highlights) rendered by shared primitives — ten-frame, clock, coins, objects, pictograph are the same grid-with-slots renderer with different skins. Ship the 4 highest-frequency types (ten-frame, clock, coins, objects) in M5; any landmark whose figure hasn't landed falls back to "prose + world sprites" by rule, not by schedule slip. drawing/trace-write answer forms are cut from Meadow Isle entirely — "set the clock hands" becomes tap-to-set (an objective interaction), no freehand input.
LowBilingual volume. Every string needs zh + en gloss, and creature names are still placeholders.
Recommended Glossary-first. Create shared/strings.ts + docs/i18n-glossary.md in M2 holding the frozen curriculum terminology (数值, 数位, 整点, 一刻…) so generator prompts and UI reuse identical wording. Creature names: one LLM naming pass (zh-first, kid-pronounceable) reviewed by you in a single sitting — 12 families, one batch, done. English gloss stays secondary exactly as QuestionView already renders it.
MediumSave migration. validateSaveState currently rejects everything except version === 1 and there is no migrator — D2 without one silently invalidates existing saves.
Recommended Land the D2 migration plan as its own PR before any stats work: migrateSave() upgrader (v1 → migrate → validate-as-v2), preservation rules (nothing dropped; player XP derived from strongest creature), Worker deploys the migrating reader before the client ships v2 writes, and fixture saves (fresh + mid-game) pinned in shared/tests/ as permanent regression tests.
MediumM3 regression bar. The versioned engine must not break the legacy numeric path that battle & shop still run on.
Recommended Golden-file equivalence test. Snapshot today's engine output for every SAMPLE_BANK question (turns, choices, judge results) into a fixture; M3's engine must reproduce it byte-for-byte through the legacy path. Add it to the existing test run so any PR that changes v1 behaviour fails immediately. New V2 forms stay additive behind the profile flag; if the bar can't be met, M3 slips rather than v1 breaking.
